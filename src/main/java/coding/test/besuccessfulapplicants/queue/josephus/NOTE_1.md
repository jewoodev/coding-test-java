# 요세푸스 문제 코드 리뷰

## 문제

N명의 사람이 원 형태로 서 있습니다. 각 사람은 1부터 N까지 번호표를 갖고 있습니다. 그리고 임의의 숫자 K가 주어졌을 때 다음과 같이 사람을 없앱니다.

- 1번 번호표를 가진 사람을 기준으로 K번째 사람을 없앱니다.
- 없앤 사람 다음 사람을 기준으로 하고 다시 K번째 사람을 없앱니다.

N과 K가 주어질 때 마지막에 살아있는 사람의 번호를 반환하는 `solution()` 함수를 구현해주세요.

### 제약조건

- N과 K는 1 이상 1000 이하의 자연수입니다.

### 입출력의 예

| N | K  | return |
| --- | --- | --- |
| 5 | 2  | 3 |

### 문제 이해

- 없앤 사람 다음 사람을 기준으로 다시 K 번째 사람을 없애는 걸 반복해야 함

---

## 제출한 풀이

```java
import java.util.ArrayDeque;

class JosephusProblem {
    public int solution(int N, int K) {
        var q = new ArrayDeque<Integer>();
        for (int i = 1; i <= N; i++) {
            q.offer(i);
        }

        int cnt = 1;
        while (q.size() != 1) {
            if (cnt % K == 0) {
                q.poll();
            } else q.offer(q.poll());
            cnt++;
        }

        return q.poll();
    }
}
```

---

## 코드 리뷰

요세푸스 문제네요. 코드 리뷰 드리겠습니다.

### 잘한 점

**`ArrayDeque` 선택이 좋습니다.** `LinkedList`보다 캐시 지역성이 좋고, `Deque`로서 `offer`/`poll`을 O(1)에 처리합니다. 코테에서 흔히 `LinkedList`를 `Queue`로 쓰는 사람이 많은데, Java의 공식 권장사항은 `ArrayDeque`입니다 (Effective Java Item 6과 JavaDoc에도 명시되어 있습니다).

**시뮬레이션 접근이 제약조건에 적절합니다.** N, K ≤ 1000이면 최악의 경우 연산량이 약 10^6 수준이므로 시뮬레이션으로 충분합니다.

### 개선할 점

#### 1. `cnt`가 불필요합니다 — 오버플로우 위험도 있습니다

현재 구조에서 `cnt`는 "K번마다 한 명 제거"를 판별하기 위한 용도인데, `cnt % K == 0`을 쓰는 순간 **매 K번째 사람을 보기 위해 큐를 앞뒤로 K-1번 순환시키고 있습니다.** 즉, `cnt`가 K번 증가할 때마다 한 명이 제거되죠.

그런데 이건 반대로 말하면, **굳이 매 턴마다 카운터를 증가시킬 필요 없이 K-1명을 한 번에 뒤로 보내고 한 명을 빼면 됩니다.**

```java
public int solution(int N, int K) {
    var q = new ArrayDeque<Integer>();
    for (int i = 1; i <= N; i++) {
        q.offer(i);
    }

    while (q.size() > 1) {
        for (int i = 0; i < K - 1; i++) {
            q.offer(q.poll());
        }
        q.poll(); // K번째 사람 제거
    }

    return q.poll();
}
```

이렇게 하면 `cnt`라는 상태 변수가 사라져서 **루프의 의도가 코드에 직접 드러납니다.** "K-1명을 뒤로 보내고 K번째 사람을 제거한다"가 그대로 읽히죠. 기존 코드는 `cnt % K == 0`을 보고 한 박자 생각해야 의도를 파악할 수 있습니다.

추가로, 기존 코드는 N, K가 크지 않아 문제는 없지만, 원칙적으로 `cnt`가 int 범위를 넘어갈 가능성을 내포합니다. 상태 변수를 없애면 이런 고민 자체가 사라집니다.

#### 2. 조건문 스타일의 일관성

```java
if (cnt % K == 0) {
    q.poll();
} else q.offer(q.poll());
```

`if`는 중괄호를 쓰고 `else`는 안 쓰는 혼재된 스타일입니다. Spring/Kotlin 컨벤션에서도 그렇지만, Java 공식 스타일 가이드에서도 **한쪽이 중괄호를 쓰면 다른 쪽도 쓰는 게 원칙**입니다. 개인 코테 풀이라 치명적이진 않지만, 리뷰어가 보면 한번 짚고 넘어갈 지점입니다.

#### 3. `size() != 1` vs `size() > 1`

의미는 같지만 `> 1`이 **의도("하나 남을 때까지 반복")를 더 직접적으로 표현**합니다. `!= 1`은 "0이 되는 경우"까지 포함하는 듯한 뉘앙스를 주는데, 실제로는 그 케이스가 발생하지 않으니 `> 1`이 더 정확합니다.

### 심화 — O(N) 수학적 해법

시뮬레이션은 O(N×K)입니다. 요세푸스 문제에는 유명한 점화식이 있습니다:

$$J(1) = 0, \quad J(n) = (J(n-1) + K) \bmod n$$

(0-indexed 기준이고, 마지막에 +1 해서 1-indexed로 변환합니다.)

```java
public int solution(int N, int K) {
    int survivor = 0; // 0-indexed
    for (int i = 2; i <= N; i++) {
        survivor = (survivor + K) % i;
    }
    return survivor + 1;
}
```

이 제약조건(N ≤ 1000)에서는 굳이 쓸 필요는 없지만, **원리를 깊이 이해하려는 성향이 있으시니** 한번 증명을 따라가 보시면 재미있을 겁니다. 핵심 아이디어는 "n명짜리 문제에서 첫 번째 사람을 제거한 후 남은 n-1명짜리 부분문제의 답을 원래 번호로 되돌리는 매핑"입니다.

### 정리

제출용으로는 지금 코드도 정답입니다. 다만 리뷰 관점에서는:

1. **`cnt` 제거 → K-1번 뒤로 보내고 1번 제거하는 구조**로 바꾸면 의도가 명확해집니다
2. `if/else` 중괄호 일관성
3. `size() > 1` 사용

이 세 가지만 반영하면 훨씬 깔끔해집니다.
